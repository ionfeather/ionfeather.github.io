RAG | 从 Demo 到进阶的完整链路

2025-12-11
阅读时长：7 分钟

做 RAG 项目的时候，很多人把 RAG 当成"向量检索 + Prompt 拼接"，但真到 production 环境里，检索不准、模型 hallucinate、上下文超长这些问题一个都不会少。

这篇笔记借一个开源 Demo 把 RAG 的完整链路串一遍，从最基本的文档处理，到生产环境里需要的评估和优化。

结论：

  1. RAG 的效果瓶颈通常不在向量检索本身，而在分块策略和重排序
  2. 混合检索（Dense + Sparse）比单一检索更稳，但融合策略比想象中重要
  3. 没有评估体系的 RAG 等于盲飞——离线指标 + 线上反馈都要抓

项目地址：Local_Pdf_Chat_RAG

环境配置

有 requirements.txt，但踩了几个小坑：

  • pip install -r requirements.txt 后，Pytorch 要求的 nvidia-cufile-cu12 和 nvidia-nvshmem-cu12 版本不对，需要手动重装
  • numpy 要降级到 2.0 以下
  • 额外装 sniffio
  • 配置 Siliconflow 和 SerpAPI 的 Key

核心模块

文档处理层

关键函数 process_multiple_pdfs，流程很标准：

  1. 清理历史数据（FAISS 向量、内容映射、BM25 索引）
  2. extract_text 提取 PDF 文本
  3. RecursiveCharacterTextSplitter 分 chunk
  4. Embedding 模型生成向量
  5. 构建 FAISS + BM25 双索引

分块策略

这个项目用的是固定长度 + overlap 的方式。但分块其实是很深的学问：

策略适用场景注意点
固定长度快速验证、通用文档可能切断句子，overlap 建议设 10%-20%
按段落/章节结构清晰的论文、报告需要 layout-aware 的 PDF 解析
Parent-Document需要兼顾检索精度和上下文完整小 chunk 检索，召回后返回完整 parent
语义分块质量要求高、预算足用 embedding 相似度判断切分点，计算成本高

这个 Demo 没做 Parent-Document，但如果是生产环境，这是性价比最高的升级之一。

向量嵌入

用的是 Bi-encoder 做 embedding，把 chunk 转成 dense vector。

选型建议：

  • 中文文档优先用 bge-large-zh 或 piccolo-base-zh，比 OpenAI 的 ada-002 对中文更友好
  • 如果需要更高精度，可以用 ColBERT（token-level 交互），但存储和计算成本高一个量级
  • MRL 向量（如 bge-m3）支持维度截断（1024→512→256），很适合需要灵活权衡精度与速度的场景
Bi-encoder vs Cross-encoder
  • Bi-encoder：查询和文档分别编码，用点积/余弦相似度比较。可以预计算文档向量，检索极快。但查询和文档没有细粒度交互，精度有天花板。
  • Cross-encoder：查询和文档拼接后一起输入模型，通过 attention 计算相关性。精度高，但无法预计算，每次都要前向传播，只能用于重排序阶段。

标准范式：Bi-encoder 做召回，Cross-encoder 做精排。

FAISS 索引

项目用的是 IndexFlatL2，暴力搜索，适合几十万以下的数据量。

索引类型原理适用规模关键参数
IndexFlatL2精确 L2 距离< 100K无
IndexIVFFlat倒排文件，先聚类再搜索100K-10Mnlist（聚类数）、nprobe（搜索类数）
IndexHNSWFlat图索引1M-100MM（连接数）、efConstruction/efSearch
IndexIVFPQIVF + 乘积量化> 10McodeSize（量化码本大小）

数据量上了百万之后，IndexFlatL2 的延迟会明显拖后腿，换成 HNSW 或 IVF 是必经之路。

BM25 索引

用 rank_bm25 + jieba 分词构建稀疏索引。

BM25 相比 TF-IDF 的两个核心改进：

  • 词频饱和：一个词出现 10 次和 100 次，相关性差距不应该线性放大。BM25 用一个饱和函数把词频增长"压扁"
  • 长度归一化：长文档天然更容易命中更多关键词，BM25 会根据文档长度对分数做惩罚
$$\text{score}(D, Q) = \sum_{i} \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{\text{avgdl}}\right)}$$

混合检索

FAISS（Dense）和 BM25（Sparse）各有盲区：

  • Dense 擅长语义泛化（“苹果”→“水果”）
  • Sparse 擅长精确关键词匹配（型号、代码、专业术语）

这个 Demo 同时用两者，就是 Hybrid Search。融合策略有两种：

  1. RRF（Reciprocal Rank Fusion）：对排名做融合，不需要分数归一化，实现简单

    $$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + r(d)}$$

  2. 线性加权：对分数做加权求和，但需要先把两种分数归一化到同一尺度

检索层

文档建好之后，用户提问时的检索流程。

查询改写

用户原始 query 往往不适合直接检索：

技术做法什么时候用
HyDE让 LLM 先写一段"理想答案"，用答案的 embedding 去检索query 太短、术语缺失
Query Expansion扩展同义词或相关概念专业领域检索
Step-back Prompting把具体问题抽象成更通用的问题，分别检索需要推理链的复杂问题
指代消解把"它"“那个"替换为具体实体多轮对话场景

HyDE 的效果取决于 LLM 对领域的了解程度。领域太专时，生成的假设文档可能偏差很大，反而引入噪声。

重排序

初排（召回）用 Bi-encoder 求快，但精度有上限。重排序阶段可以用 Cross-encoder 做一次精排：

from sentence_transformers import CrossEncoder
reranker = CrossEncoder('BAAI/bge-reranker-large')

pairs = [[query, doc] for doc in retrieved_docs]
scores = reranker.predict(pairs)

实际部署时的 trade-off：通常只重排 Top-20 或 Top-50，全部重排延迟太高。

生成层

检索到片段后，把上下文和 query 拼成 Prompt 传给 LLM。

Prompt 设计

基础的 RAG Prompt 模板：

你是专业的问答助手。请根据以下参考资料回答问题。
如果资料中没有相关信息，请明确说明"根据现有资料无法回答"。

参考资料：
{context}

用户问题：{question}

请回答：

关键设计：

  • 给模型明确的"拒绝回答"许可，降低 hallucinate 概率
  • 要求标注引用来源（“根据文档[1]…"）
  • 必要时加 few-shot 示例，展示"如何引用"和"如何拒绝”

Lost in the Middle

LLM 对 Prompt 中间部分的信息敏感度更低，容易忽略。缓解方法：

  • 把最相关的 chunk 放在开头和结尾
  • 控制总上下文长度，避免信息淹没

上下文长度管理

检索到多个 chunk 后，总长度可能超过上下文窗口：

策略做法适用场景
截断只保留前 N 个 chunk最简单，可能丢失后面相关信息
Map-Reduce每个 chunk 单独生成中间答案，再汇总长文档问答
Refine逐个 chunk 迭代修正答案需要逐步精化的场景
摘要压缩先用 LLM 把多个 chunk 压缩成短摘要chunk 数量极多时

高级架构

这个 Demo 是 Naive RAG，但 2024-2025 年涌现了不少更成熟的架构。

★ Self-RAG

让模型自己决定是否需要检索、检索多少次：

  • 模型生成时遇到不确定的内容，输出 [Retrieve] token
  • 触发检索后，用 [IsSupp] 判断检索结果是否支持当前陈述
  • 不支持则修正或重新检索

需要在训练阶段引入这些特殊 token，让模型学会动态决策。

★ RAPTOR

解决跨 chunk 的宏观语义检索。比如一本书分散在几十个 chunk 中，问"这本书的核心观点是什么”，传统 RAG 很难回答。

RAPTOR 的做法：

  1. 底层：原始 chunk 的 embedding
  2. 中层：相邻 chunk 聚类，LLM 生成摘要，再做 embedding
  3. 顶层：对中层摘要继续聚类、摘要，形成树状结构
  4. 检索时既能在底层找细节，也能在顶层找宏观结论

★ GraphRAG

把文档转化为知识图谱，节点是实体，边是关系。

和向量 RAG 的互补性：

  • 向量 RAG 擅长"相似语义"匹配
  • GraphRAG 擅长"结构化关系"推理（如"A 公司的 CEO 之前任职的公司是什么"）

评估体系

没有评估的 RAG 等于盲飞。

检索评估

指标含义计算
Hit Rate@K正确答案是否在 Top-K 中二元判断取平均
MRRMean Reciprocal Rank$\frac{1}{|Q|} \sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}$
Recall@K相关文档被召回的比例召回的相关文档数 / 总相关文档数
Precision@KTop-K 中相关文档的比例Top-K 中的相关文档数 / K

生成评估

指标评估什么
Faithfulness回答是否忠于检索内容
Answer Relevance回答是否切题
Context Precision检索的上下文中有多少是相关的
Answer Correctness回答的事实正确性

可以用 RAGAS 做自动化评估。

我的判断

如果让我基于这个 Demo 搭一个 production-ready 的 RAG，升级优先级会是：

  1. Parent-Document Retrieval（性价比最高，小改动大提升）
  2. 加一层 Cross-encoder Reranker（Bi-encoder 召回 + Cross-encoder 精排是标准范式）
  3. Query Rewriting（HyDE 或简单的指代消解）
  4. 评估体系（先用 RAGAS 跑起来，再逐步加线上反馈）
  5. 索引升级（数据量大了之后换 HNSW 或 IVF）

至于 Self-RAG、RAPTOR、GraphRAG 这些高级架构，它们解决的是特定场景的问题（动态检索、跨文档推理、关系推理），不是每个系统都需要的。先跑通基础链路，有明确瓶颈了再引入，否则容易过度设计。
